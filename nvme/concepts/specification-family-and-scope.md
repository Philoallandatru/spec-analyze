# 规范族与适用范围（Specification Family and Scope）

## 概述

NVMe 规范并非单一文档，而是一个相互协作的**规范族**（Specification Family）。这个规范族包含多个独立的规范文档，每个规范负责不同的功能领域，共同构成了完整的 NVMe 技术体系。

理解规范族的结构对于正确使用 NVMe 规范至关重要。本文档介绍各规范之间的关系、适用范围以及它们如何协同工作。

## 规范族的组成

NVMe 规范族包含以下几类规范文档：

### 1. 基础规范（Base Specification）

**核心作用**：定义主机（Host）软件与 NVM 子系统之间通信的传输无关协议（transport-independent protocol）。

基础规范是整个 NVMe 规范族的核心，它定义了：
- 主机与 NVM 子系统之间的通信协议
- 控制器（Controller）的基本属性和行为
- 所有 I/O 命令集共有的基础概念
- 与具体传输方式无关的抽象接口

### 2. I/O 命令集规范（I/O Command Set Specification）

**核心作用**：在基础规范之上扩展具体的命令和数据结构。

不同的存储介质和应用场景需要不同的命令集。I/O 命令集规范定义了：
- 针对特定应用场景的命令（如 NVM、Key-Value、Zoned Namespace 等）
- 这些命令相关的数据结构
- 命令的具体语义和行为

### 3. 传输规范（Transport Specification）

**核心作用**：将协议和控制器属性绑定到具体的传输介质上。

传输规范定义了如何在具体的物理传输介质上实现 NVMe 协议，包括：
- PCIe（最常见的 NVMe 传输方式）
- RDMA（用于网络存储）
- TCP（用于 IP 网络）
- Fibre Channel（用于光纤通道网络）

### 4. NVMe-MI 规范（NVMe Management Interface）

**核心作用**：针对设备管理场景定义的接口规范。

NVMe-MI 提供了带外管理功能，允许管理软件在不通过主机 NVMe 驱动的情况下管理 NVMe 设备。

### 5. 引导规范（Boot Specification）

**核心作用**：针对系统引导场景定义的特殊要求。

引导规范定义了如何使用 NVMe 设备作为系统启动盘，包括 UEFI 和传统 BIOS 环境下的引导支持。

## 基础规范的边界

### 规范范围

基础规范明确定义了哪些内容属于其管辖范围，哪些内容被有意排除在外。这种明确的边界定义确保了接口的清晰性和实现的灵活性。

| 类别 | 说明 | 具体内容 |
|------|------|----------|
| **范围之内** | 规范明确定义的内容 | • 控制器（Controller）的属性和行为<br>• 管理命令和 I/O 命令<br>• 所有 I/O 命令集的共有特性<br>• 主机与控制器的交互协议 |
| **传输绑定** | 通过特定传输方式实现 | • 控制器通过具体的 NVMe 传输方式实现<br>• 不同传输方式的具体实现细节由传输规范定义 |
| **使用模型** | 不做规定的内容 | • SSD、主内存、缓存、备份、冗余等具体用途<br>• 这些应用场景超出规范范围，由实现者自行决定 |
| **实现细节** | 被抽象化的内容 | • 存储介质管理策略<br>• 磨损均衡（Wear Leveling）算法<br>• 擦除策略<br>• 缓存（Cache）算法和策略 |

### 设计理念

这种边界划分有重要的设计意义：

**接口与实现分离**：通过将实现细节抽象化，NVMe 规范允许各种不同的实现方式：
- **物理设备**：真实的 SSD 硬件
- **网关设备**：将 NVMe 协议转换为其他协议的设备
- **软件实现**：虚拟化环境中的软件 NVMe 控制器

所有这些不同的实现只要遵循相同的协议契约，就能保证互操作性。

## 需求语言规范

### 关键字定义

阅读规范时，需要正确理解各种需求关键字的含义。NVMe 规范使用特定的关键字来表达不同程度的要求：

| 关键字 | 含义 | 遵守级别 | 说明 |
|--------|------|----------|------|
| **mandatory** / **shall** | 必须、强制 | 强制性要求 | 为了确保一致性（conformance）和互操作性（interoperability），**必须**遵守的要求 |
| **may** | 可以、允许 | 允许的选择 | 表示允许的行为，规范对此不表达偏好 |
| **optional** | 可选的 | 可选功能 | 不是必需的功能，但如果实现了，则必须符合规范定义的行为 |
| **should** | 应该、建议 | 强烈建议 | 强烈倾向的选择，除非有充分理由，否则应该遵循 |
| **obsolete** | 已废弃 | 历史兼容 | 早期版本中存在但已在当前版本中移除的功能 |

### 保留值的处理规则

规范中经常会遇到"保留"（Reserved）的字段和编码值。正确处理保留值对于实现兼容性至关重要：

#### 重要原则

**保留的存储空间和编码值不是普通的扩展点**。它们是为未来规范版本预留的，当前实现必须按以下规则处理：

| 场景 | 处理规则 | 违反后果 |
|------|----------|----------|
| **写入保留字段** | 必须将保留字段清零（写入 0） | 未来版本可能赋予这些字段新的含义，非零值可能导致不可预期的行为 |
| **读取保留字段** | 忽略保留字段的值，不依赖其内容 | 保留字段的值是未定义的，不能用于任何逻辑判断 |
| **收到保留编码值** | 在已定义的命令字段中收到保留编码值应视为错误 | 这表明发送方可能来自未来版本或存在实现错误 |
| **向控制器写入保留编码** | 结果是未定义的 | 控制器可能拒绝、忽略或产生不可预期的行为 |

#### 示例说明

假设某个字段定义为：
- 位 [3:0]：定义为命令类型（0h = Read, 1h = Write, 2h-Fh = Reserved）
- 位 [7:4]：保留

正确的处理方式：
- **主机软件**：只能使用 0h（Read）或 1h（Write），必须将位 [7:4] 设置为 0
- **控制器**：如果收到位 [3:0] 为 2h-Fh 的值，应返回错误状态

## 使用建议

在使用 NVMe 规范时，建议按以下顺序理解：

1. **先理解基础规范**：掌握 NVMe 的核心概念和通用协议
2. **选择适合的传输规范**：根据应用场景选择 PCIe、RDMA、TCP 等
3. **选择需要的 I/O 命令集**：根据存储需求选择 NVM、KV、ZNS 等命令集
4. **按需参考 NVMe-MI 和引导规范**：根据管理和引导需求决定是否需要这些规范
