# 命名空间（Namespace）

## 为什么需要命名空间？

想象一下，你买了一个 1TB 的 SSD，但你想：
- 把 500GB 分给 Windows 系统使用
- 把 300GB 分给 Linux 虚拟机使用
- 把 200GB 留作备份

传统硬盘通过"分区"来实现这个需求，NVMe 用的是更灵活的**命名空间**。

**类比理解：**
- 整个 SSD 就像一栋大楼
- 命名空间就像大楼里的独立房间
- 每个房间可以有不同的租户（操作系统、应用程序）
- 房间可以动态创建、删除、调整大小

## 命名空间是什么？

**简单定义：** 命名空间是一段被格式化为逻辑块的存储空间，主机可以通过控制器来读写它。

**更通俗地说：** 命名空间就是你能看到和使用的"虚拟硬盘"。

### 为什么叫"命名空间"？

- **命名**：每个命名空间都有一个唯一的编号（Namespace ID）
- **空间**：它代表一段存储空间

就像你家的房间有门牌号，命名空间也有 ID（1、2、3...）。

## 核心概念：三个独立的关系

理解命名空间，必须搞清楚三个问题。这三个问题**互相独立**，不能混为一谈：

### 1. 是否已分配？（Allocation）

**问题：** 这段存储空间被"圈"出来了吗？

```text
NVM 存储资源 ──── 分配 ───→ 命名空间
    1TB            500GB      Namespace 1
```

**类比：** 就像从一块大地上划出一块地，准备盖房子。地划出来了，但房子还没盖。

**关键点：**
- 分配后，这段存储资源被保留了
- 但还不能使用，需要进一步"挂载"

### 2. 是否已挂载？（Attachment）

**问题：** 命名空间连接到控制器了吗？

```text
控制器 A ──── 挂载 ───→ Namespace 1
控制器 B ──── 挂载 ───→ Namespace 1 (可以共享)
```

**类比：** 房子盖好了，但还需要安装门和钥匙。挂载就是给控制器配钥匙。

**关键点：**
- 命名空间可以挂载到一个或多个控制器
- 没挂载到某个控制器，那个控制器就看不到这个命名空间
- 挂载关系可以动态改变

### 3. 是否可访问？（Visibility）

**问题：** 主机能通过某条路径访问到吗？

```text
主机路径 ──── 可见性 ───→ 命名空间
```

**类比：** 你有钥匙（挂载），但还需要一条路通到房子（网络路径、PCIe 连接）。

**关键点：**
- 网络存储中，即使挂载了，网络断了也访问不到
- 多路径情况下，可能有多条路到达同一个命名空间

## 完整流程示例

让我们看一个完整的例子，理解这三个步骤：

```text
步骤 1：分配命名空间
    管理员：从 1TB SSD 中分配 500GB
    结果：创建了 Namespace 1，但还不能用

步骤 2：挂载到控制器
    管理员：将 Namespace 1 挂载到控制器 A
    结果：控制器 A 现在知道有 Namespace 1 这个"房间"

步骤 3：主机访问
    主机：通过控制器 A 发送读写命令
    结果：可以读写 Namespace 1 的数据了
```

### 常见场景

**场景 1：私有命名空间**
```text
Namespace 1 只挂载到控制器 A
    → 只有通过控制器 A 的主机能访问
    → 典型用法：个人电脑的 SSD
```

**场景 2：共享命名空间**
```text
Namespace 1 同时挂载到控制器 A 和控制器 B
    → 多个主机可以同时访问
    → 典型用法：共享存储、虚拟机迁移
```

**场景 3：未挂载的命名空间**
```text
Namespace 2 已分配，但未挂载到任何控制器
    → 存储空间已保留，但暂时不使用
    → 典型用法：预留空间、动态分配
```

## 命名空间写保护机制

### 为什么需要写保护？

有些数据非常重要，不能被意外修改或删除：
- 系统固件
- 审计日志
- 归档数据

命名空间写保护提供了四个级别的保护：

### 保护级别详解

| 级别 | 说明 | 能否解除？ | 使用场景 |
|------|------|-----------|---------|
| **无保护** | 正常读写 | - | 日常使用的数据盘 |
| **写保护** | 只读，但可以随时解除 | ✅ 可以 | 临时保护，防止误操作 |
| **断电前保护** | 只读，直到断电重启才能解除 | ❌ 断电前不能 | 重要配置文件，重启前不能改 |
| **永久保护** | 只读，永远不能解除 | ❌ 永远不能 | 固件、审计日志等不可变数据 |

**特性 ID：** `84h`

[规范 PDF 第 425-426 页](../_source/pages/page-425.md)

### 重要行为

当命名空间进入"断电前保护"或"永久保护"状态时：

```text
系统自动执行：
1. 将命名空间的所有缓存数据刷新到存储芯片
2. 将元数据也刷新到存储芯片
3. 确保数据完全持久化后，才生效保护
```

**为什么这样做？** 避免保护后发现数据还在缓存里，断电就丢失了。

## 术语解释

### 什么是"逻辑块"（Logical Block）？

**简单理解：** 存储设备的最小读写单位。

- **类比：** 就像书本的"页"，你不能读半页，只能一页一页地读
- **大小：** 通常是 512 字节或 4096 字节（4KB）
- **编号：** 每个逻辑块有个编号，叫 LBA（Logical Block Address）

### 什么是"非易失性存储"？

**简单理解：** 断电后数据不丢失的存储。

- **易失性存储**：内存（RAM），断电数据就没了
- **非易失性存储**：SSD、硬盘，断电数据还在

### 什么是"元数据"（Metadata）？

**简单理解：** 关于数据的数据。

**例子：**
- **数据**：一个文件的内容
- **元数据**：文件名、大小、创建时间、权限等

在命名空间中，元数据包括：
- 逻辑块的保护信息
- 错误检测码
- 访问权限

## 新手常见疑问

### Q1: 命名空间和分区有什么区别？

**A:** 
| 特性 | 传统分区 | NVMe 命名空间 |
|------|---------|-------------|
| **创建方式** | 需要重新格式化，数据会丢失 | 可以动态创建，不影响现有数据 |
| **数量限制** | 通常最多 4 个主分区 | 可以有很多个（理论上 2³² 个） |
| **大小调整** | 困难，通常需要第三方工具 | 相对简单，支持在线调整 |
| **共享** | 不支持 | 支持多个控制器共享 |

### Q2: 我的 SSD 有几个命名空间？

**A:** 
- **大部分消费级 SSD**：只有 1 个命名空间（Namespace 1）
- **企业级 SSD**：可以有多个，用于虚拟化、多租户等场景
- **如何查看**：在 Linux 上用 `nvme list` 命令，在 Windows 上用磁盘管理工具

### Q3: 命名空间和卷（Volume）是一回事吗？

**A:** 不完全是：
- **命名空间**：NVMe 设备层面的概念，由硬件和驱动管理
- **卷（Volume）**：操作系统层面的概念，由文件系统管理

```text
物理层次：
NVMe SSD → 命名空间 → 文件系统 → 卷/分区 → 目录和文件
```

一个命名空间上可以创建多个操作系统卷。

### Q4: 为什么说"分配、挂载、可访问"是独立的？

**A:** 看这个例子：

```text
情况 1：已分配，已挂载，可访问
    → 正常使用状态

情况 2：已分配，已挂载，不可访问
    → 网络断了，或者主机没连接到控制器

情况 3：已分配，未挂载，不可访问
    → 命名空间存在，但没有配置给任何控制器使用

情况 4：未分配
    → 存储空间没有被用来创建命名空间
```

这三个状态互相独立，所以有 8 种组合（2³）。
