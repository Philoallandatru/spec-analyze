# 分散命名空间生命周期（Dispersed Namespace Lifecycle）

## 概述

分散命名空间（Dispersed Namespace）是一种特殊的逻辑命名空间（Logical Namespace），允许主机（Host）通过两个或多个 NVM 子系统（NVM Subsystem）中的控制器（Controller）同时访问同一个命名空间。这一特性支持在线迁移和数据复制，而无需在 Base 规范中引入跨子系统的分散机制。

**核心价值**：
- 支持无中断的在线迁移
- 实现跨子系统的数据复制
- 提供多路径并发访问能力
- 简化存储系统的扩展和维护

## 概念模型

为了便于理解分散命名空间的工作原理，我们可以用以下模型来描述：

```text
                         稳定的全局身份标识
                    （相同的 NGUID 或 UUID 值）
                                |
主机路径 --> [NVM 子系统 A: NSID x] === [NVM 子系统 B: NSID y]
                    |      源/副本      |
                    `-- NSID 仅在本地有效 --'
```

**关键理解要点**：

1. **全局身份 vs 本地标识**：每个分散命名空间都有一个全局唯一的标识符（NGUID 或 UUID），但在不同的 NVM 子系统中，它可能对应不同的本地命名空间 ID（NSID）

2. **主机视角统一**：尽管底层有多个命名空间实例分布在不同子系统中，但主机将它们视为同一个命名空间

3. **多路径访问**：主机可以通过连接到不同参与子系统的控制器来访问同一个逻辑命名空间

这一模型保留了规范图 611-614 中的关键区分：主机可以通过一个或多个参与子系统访问，而分散命名空间则横跨它们各自的本地命名空间实例。

## 典型使用场景

分散命名空间支持多种使用模式，每种模式都有其特定的路径行为和存储行为：

| 使用模式 | 主机路径行为 | 存储行为 | 典型应用场景 |
| ------- | ----------- | -------- | ----------- |
| **多路径并发访问** | 主机可通过多个参与子系统的控制器同时访问同一全局命名空间 | 各参与方独立持有该命名空间实例，通过全局标识符与 ANA 在主机视角下统一呈现 | 高可用性应用、负载均衡 |
| **在线迁移** | 先添加目标路径，再移除源路径，迁移过程中路径共存 | 数据从源逐步移动到目标，I/O 操作持续可用，无需停机 | 存储升级、数据中心迁移 |
| **共享转分散** | 原本挂载到同一子系统多个控制器的共享命名空间切换到跨子系统模式 | 各参与子系统保留本地实例，并通过主机标识符（Host Identifier）进行全局聚合 | 存储架构演进 |
| **单参与方降级** | 分散命名空间退化为仅通过单个参与方访问 | 其他参与方的关联解除，但 NSID 和全局 ID 保持不变，后续可重新扩展 | 故障恢复、维护模式 |

**说明**：以上是规范定义的主要场景。在实际部署中，到子系统的连线可能代表到多个控制器的多条物理或逻辑连接。

## 管理与生命周期操作

### 创建与挂载

命名空间管理（Namespace Management）命令可以创建一个支持跨子系统并发访问的命名空间。创建后，可以通过命名空间挂载（Namespace Attachment）命令独立地将该命名空间挂载到特定控制器，或从控制器上卸载。

### 转换机制

- **共享到分散的转换**：共享命名空间（Shared Namespace）可以转换为分散命名空间
- **私有命名空间的限制**：私有命名空间（Private Namespace）不应直接转换为分散命名空间
- **转换通知**：无论是哪个方向的转换，系统都会报告"命名空间属性已更改"（Namespace Attribute Changed）事件

**注意**：将命名空间分散或转换的具体实现机制超出了 NVMe 规范的定义范围，由各厂商自行实现。

### 删除操作

通过命名空间管理（Namespace Management）或容量管理（Capacity Management）命令删除命名空间时：

- 删除操作作用于处理该命令的参与子系统
- 删除是否会传播到其他参与方取决于具体实现
- 建议在删除前仔细评估对其他参与方的影响

[PDF p. 534](../_source/pages/page-534.md)

## 身份标识约定

### NSID 的作用域

命名空间 ID（NSID, Namespace ID）在同一个参与 NVM 子系统内的不同控制器之间是一致的，但跨参与子系统时可能不同。这意味着：

- **子系统内一致**：同一子系统内的所有控制器看到相同的 NSID
- **跨子系统可变**：不同子系统中，同一个分散命名空间可能使用不同的 NSID

### 全局标识符

为了跨子系统关联同一个分散命名空间，必须使用全局唯一的命名空间标识符：

| 标识符类型 | 优先级 | 用途 | 说明 |
| --------- | ----- | ---- | ---- |
| **NGUID** | 最高 | 优先用于跨子系统关联 | Namespace Globally Unique Identifier，如果存在则必须使用 |
| **UUID** | 次优 | 当 NGUID 不存在时使用 | Universally Unique Identifier |
| **EUI64** | 不适用 | **绝不**作为分散命名空间的跨子系统标识符 | Extended Unique Identifier 64-bit，仅用于其他场景 |

**关键规则**：全局标识符必须在所有参与子系统中保持相同值，这是主机识别和关联分散命名空间的基础。

[PDF pp. 534-535](../_source/pages/page-534.md)

## 能力发现与主机声明

### 发现分散命名空间能力

主机通过以下机制发现和使用分散命名空间：

1. **能力发现**：通过 `NMIC.DISNS` 字段（Namespace Management and Attachment Feature, Dispersed Namespace Support）发现控制器是否支持分散命名空间

2. **参与方发现**：通过"分散命名空间参与 NVM 子系统日志"（Dispersed Namespace Participating NVM Subsystems Log Page）查询有哪些子系统参与了该分散命名空间

3. **命令集兼容性**：每个提供访问的控制器必须支持该命名空间关联的 I/O 命令集（I/O Command Set）

[PDF p. 535](../_source/pages/page-535.md)

### 主机行为声明

主机需要通过"主机行为支持"（Host Behavior Support）特性明确声明其对分散命名空间的支持：

**`HDISNS` 标志（Host Dispersed Namespace Support）**：

- **设置为 `1h`**：主机承诺将跨控制器和参与子系统，把具有相同全局标识符的命名空间视为同一个逻辑命名空间，并使用一个在所有参与方之间全局唯一的主机标识符（Host Identifier）

- **设置为 `0`（未声明支持）**：参与方可以采取以下策略之一：
  - 仅在主机能通过单个子系统访问该分散命名空间时允许访问
  - 完全禁止对该分散命名空间的访问
  - 在检测到主机连接到第二个参与子系统时拒绝 Connect 命令（这会导致该主机失去对该子系统上所有命名空间的访问）
  - 接受 Connect 命令，但仅屏蔽对该分散命名空间的访问，允许访问其他命名空间

[PDF pp. 535-536](../_source/pages/page-536.md)

### 访问限制与命令约束

当分散访问被禁止时（`HDISNS=0` 且控制器拒绝分散访问），以下命令会失败：

**失败的命令类型**：

1. **I/O 命令**：指定分散命名空间 NSID 的所有 I/O 命令
2. **特定管理命令**：规范中列出的特定管理命令
3. **广播命令**：如果作用域包含分散命名空间，则广播的 Set Features 命令会被禁止
4. **格式化命令**：宽作用域的 Format NVM 命令（如果作用域包含分散命名空间）

**错误码**：这些命令将返回"主机分散命名空间支持未启用"（Host Dispersed Namespace Support Not Enabled）错误状态。

[PDF pp. 536-537](../_source/pages/page-536.md)

## 多路径与 ANA 支持

对于同时暴露同一分散命名空间的所有参与方，应支持**非对称命名空间访问**（ANA, Asymmetric Namespace Access）特性。通过 ANA，主机可以：

- 了解通过不同路径访问命名空间的优化程度
- 根据路径状态（优化、非优化、不可访问等）做出智能的路径选择
- 在路径状态变化时动态调整 I/O 路由

**路径关联**：使用全局标识符（NGUID 或 UUID）来关联跨参与方的同一命名空间的不同路径。

[PDF p. 537](../_source/pages/page-537.md)

## Reservation 预留协调

### 跨子系统预留模型

分散命名空间中的预留（Reservation）需要在所有参与方之间进行协调。以下模型展示了预留信息的呈现方式：

```text
主机标识符（在所有参与方之间全局唯一）
        v                         v
[参与方 A: 本地 CNTLID]  [参与方 B: 远端注册方]
        `---- Reservation Report -+--> CNTLID = FFFDh
```

**关键理解**：

- 本地控制器使用其实际的 CNTLID（Controller ID）
- 远程参与方上的同一主机的一个或多个控制器会被合并为单个注册方记录
- 远程注册方使用哨兵值 `CNTLID=FFFDh` 来表示

[PDF p. 537](../_source/pages/page-537.md)

### 预留命令支持

主机和控制器需要协同支持分散命名空间上的预留操作：

**主机声明**：

主机通过在以下预留命令中设置 `DISNSRS=1` 标志（Dispersed Namespace Reservation Support）来声明支持远端控制器哨兵值：

- Reservation Acquire（获取预留）
- Reservation Register（注册预留）
- Reservation Release（释放预留）
- Reservation Report（预留报告）

**控制器要求**：

支持分散命名空间预留能力的控制器必须接受该标志位。

**错误处理**：

如果主机已声明 `HDISNS=1`（支持分散命名空间），但在预留命令中设置 `DISNSRS=0`（不支持分散预留），则发往分散命名空间的预留命令将失败，返回"命名空间是分散的"（Namespace Is Dispersed）错误状态。
