# 传输模型（Transport Models）

## 概述

NVMe 规范定义了主机（Host）与 NVM 子系统（NVM Subsystem）之间的两种基本通信模型：**基于内存的传输**（Memory-Based Transport）和**基于消息的传输**（Message-Based Transport）。传输层（Transport Layer）负责决定命令、响应和数据如何在主机和存储子系统之间传递。

这两种模型的选择取决于底层的物理连接方式和传输协议。[PDF pp. 39-46](../_source/pages/page-039.md)

## 两种传输模型的核心差异

### 基于内存的传输（Memory-Based）

**典型代表：PCIe**

在这种模型中，主机和控制器共享内存空间。控制器可以直接访问主机内存中的提交队列（SQ）、完成队列（CQ）和数据缓冲区。通信通过内存映射 I/O（MMIO）和直接内存访问（DMA）完成。

```text
基于内存的传输 (例如 PCIe)

主机内存 + MMIO
   SQ / CQ / 数据
        |
        v
     控制器
```

### 基于消息的传输（Message-Based）

**典型代表：NVMe over Fabrics（TCP、RDMA 等）**

在这种模型中，主机和控制器通过网络结构（Fabric）交换封装的消息。命令和响应被打包成"胶囊"（Capsule），通过网络协议传递。

```text
基于消息的传输 (例如 Fabrics)

主机端                           子系统端
  |  命令胶囊 ──────────────→    |
  |  ←────────────── 响应胶囊    |
  |  ←──────────────→ 数据       |
```

基于消息的传输还可以进一步细分为两种子类型，区分标准是传输过程中是否有显式内存操作参与：
- **纯消息模式**（Message-Only）：所有数据都通过消息传递
- **消息/内存混合模式**（Message/Memory）：消息传递命令，但数据可能通过显式内存操作传输

[PDF p. 39](../_source/pages/page-039.md)

## NVMe 传输架构分层

NVMe 采用了清晰的分层架构，将传输无关的核心协议与具体的传输实现分离：

```text
┌─────────────────────────────────────────────────────┐
│  NVMe 核心架构                                       │
│  (队列机制、命令集、控制器属性等)                    │
└─────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  传输绑定层 (Transport Binding)                      │
│  (传输特定的适配和服务)                              │
└─────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  NVMe 传输层 (NVMe Transport)                        │
└─────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────┐
│  底层网络协议 / 物理网络结构                         │
│  (不属于 NVMe 规范范围)                              │
└─────────────────────────────────────────────────────┘
```

**分层说明：**
- **NVMe 核心架构**：定义了与传输方式无关的通用接口和行为
- **传输绑定层**：将 NVMe 协议映射到具体的传输实现
- **NVMe 传输层**：在底层网络之上提供可靠的传输服务
- **底层协议**：如 PCIe、RDMA、TCP 等，这些协议本身不属于 NVMe 规范族

这种分层设计确保传输无关的 NVMe 接口位于绑定层之上、原生网络结构（Fabric）之下；原生网络服务不在 NVMe 规范族的范围内。[PDF pp. 43-44](../_source/pages/page-043.md)

## 规范文档的组织结构

NVMe 规范族包含多个文档：

- **基础规范（Base Specification）**：有意涵盖两种传输模型的通用部分
- **各传输规范**：将 NVMe 协议（包括控制器属性）绑定到具体传输，例如：
  - NVMe over PCIe
  - NVMe over RDMA
  - NVMe over TCP

[PDF p. 23](../_source/pages/page-023.md)

## 胶囊（Capsule）机制详解

在 NVMe over Fabrics 中，信息交换的基本单位是**胶囊（Capsule）**。

### 什么是胶囊？

胶囊是一个封装单元，可以包含：
- 一条命令或一个响应
- 可选的命令/响应数据
- 散布/聚集列表（SGL，Scatter-Gather List）

[PDF p. 29](../_source/pages/page-029.md)

## 传输层与端口（Port）

### NVMe 传输层的职责

NVMe 传输层在底层网络结构之上提供以下服务：
- **可靠交付**：确保命令和数据正确传递
- **传输无关性**：与底层物理互联和协议无关

### NVM 子系统端口

**NVM 子系统端口**（NVM Subsystem Port）是一个协议接口，具有以下特性：

- 可以聚合一个或多个物理网络接口
- 通过 **NQN**（NVMe Qualified Name）和 **16 位端口 ID** 进行标识
- 一个端口可以支持多种传输协议
- 已连接的控制器会绑定到特定的子系统端口

[PDF pp. 39, 44](../_source/pages/page-039.md) [PDF p. 44](../_source/pages/page-044.md)

## 三种传输模型对比

| 传输模型 | 命令/响应传递方式 | 数据传输方式 |
|---------|-----------------|-------------|
| **基于内存** | 显式内存读/写操作 | 显式内存读/写操作 |
| **纯消息** | 胶囊/消息 | 仅通过胶囊/消息 |
| **消息/内存混合** | 胶囊 | 胶囊或显式内存操作 |

两种基于消息的子类型共享基于胶囊的命令和响应传输方式，区别在于数据如何移动。[PDF pp. 39-40](../_source/pages/page-039.md)

## 基于消息的队列机制

对于基于消息的队列，传输层以绑定特定的方式交付命令胶囊并报告其到达。

### 胶囊大小的确定

- **管理队列胶囊大小**：由传输绑定（Binding）固定
- **I/O 队列胶囊大小**：通过 Identify Controller 命令报告
  - 包括 I/O 命令胶囊的大小
  - 包括 I/O 响应胶囊的大小

### SGL 与数据传输

管理队列和 I/O 队列均支持基于 **SGL**（Scatter-Gather List，散布/聚集列表）的传输。传输绑定定义了哪些 SGL 类型可以选择：
- **胶囊内数据**（In-Capsule Data）：数据直接包含在胶囊中
- **显式内存事务**：通过内存操作单独传输数据

**重要规则：** 胶囊中未使用且非保留的字节是未定义的（Undefined），不得被解释。

[PDF p. 110](../_source/pages/page-110.md)

## Fabrics 胶囊结构与数据放置

### 命令胶囊结构

```text
命令胶囊 (主机 → 控制器)
字节 0                 63 64                              N-1
┌────────────────────────┬──────────────────────────────┐
│  64 字节 SQE           │  数据 或 额外的 SGL           │
│  (提交队列条目)        │                               │
└────────────────────────┴──────────────────────────────┘
```

### 响应胶囊结构

```text
响应胶囊 (子系统 → 主机)
字节 0         15 16                                     N-1
┌──────────────────┬────────────────────────────────────┐
│  16 字节 CQE     │  可选数据                           │
│  (完成队列条目)  │                                     │
└──────────────────┴────────────────────────────────────┘
```

**关键要点：**
- 命令胶囊在 SQE 之后**不会同时混合**数据和额外的 SGL
- SQE 中的**命令标识符**（Command Identifier）在该队列的所有未完成命令中必须唯一
- 响应 CQE 与之前的某个命令胶囊一一对应

该布局基于规范中的 Figures 75-76 重建。[PDF p. 111](../_source/pages/page-111.md)

### 数据放置策略

| SGL 选择的数据放置方式 | 胶囊处理结果 |
|---------------------|-------------|
| **内存地址** | 数据通过传输支持的内存事务传输（如 RDMA） |
| **偏移量** | 数据直接承载在命令或响应胶囊中 |
| **额外描述符段** | 描述符紧接 SQE 之后连续排列 |

### 胶囊内数据布局细节

SQE 包含一个 SGL 条目。对于胶囊内数据：
- **额外的胶囊内 SGL**：从字节 64 开始
- **胶囊内命令数据**：从字节偏移量 `64 + ICDOFF × 16` 开始
- 中间的任何字节都是未定义的

### 错误处理

描述符的数量和空间限制来自 Identify Controller：
- 违反描述符数量限制：命令会被中止，返回 **Invalid Number of SGL Descriptors** 错误
- SGL 偏移量无效：返回 **SGL Offset Invalid** 错误

这些布局和限制已与 PDF 规范的第 111-114 页进行了核对。[PDF pp. 111-114](../_source/pages/page-111.md)

## 关键要点总结

1. **两种基本模型**：NVMe 支持基于内存和基于消息两种传输模型，分别适用于本地连接和网络存储场景
2. **细分类型**：基于消息的传输进一步分为纯消息和消息/内存混合两种子类型
3. **分层架构**：核心 NVMe 协议与具体传输实现通过绑定层清晰分离
4. **胶囊机制**：Fabrics 传输使用胶囊封装命令、响应和数据
5. **灵活数据放置**：通过 SGL 支持多种数据放置策略，平衡性能和网络效率
6. **可扩展性**：统一的架构允许 NVMe 支持从 PCIe 到各种网络传输协议
