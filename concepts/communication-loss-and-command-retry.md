# 通信丢失与命令重试

## 概述

在 NVMe 系统中，主机（Host）与控制器（Controller）之间的通信链路可能会意外中断。当通信丢失发生时，主机面临一个关键问题：**那些已经提交但尚未收到完成响应的命令（Outstanding Commands）究竟执行了没有？**

安全恢复的核心理念并非"越快重试越好"，而是遵循一个严谨的流程：
1. **先建立边界**：确认控制器已经不再处理原命令
2. **再分类处理**：根据命令的幂等性（Idempotence）和状态可观测性决定恢复策略

本文档详细说明了 NVMe 规范中关于通信丢失检测、命令停止保证和重试分类的完整机制。

**规范依据**: [PDF pp. 691-694](../_source/pages/page-691.md)

## 恢复流程心智模型

下图展示了从通信丢失到安全重试的完整路径：

```text
[通信丢失发生]
       |
       v
终止关联（Association）和消息传输（Message Transport）
       |
       v
等待控制器检测到丢失：2×KATT / 3×KATT / 传输层通知
       |
       v
等待命令队列超时（CQT）或实现特定的停止时间
       |
       v
[控制器不再可能处理原命令]
       |
       +-------------+------------------+-------------------+
       |             |                  |
  无限制重试    延迟重试          状态依赖重试
  (立即执行)    (等待后执行)      (需上层决策)
```

**流程说明**：
- **阶段1**：断开传输连接，阻止新的命令交互
- **阶段2**：等待控制器侧超时检测机制触发
- **阶段3**：等待已提交命令完全停止处理
- **阶段4**：根据命令特性选择恢复策略

**规范依据**: [PDF pp. 691-694](../_source/pages/page-691.md)

## 错误类型与恢复契约

### 错误分类

NVMe 定义了多种错误严重程度，需要不同的恢复手段：

| 错误类型 | 影响范围 | 恢复方法 |
|---------|---------|---------|
| **普通命令错误** | 单个命令失败 | 检查 CQE 中的 Retry 位，决定是否重试 |
| **严重队列错误** | 整个提交队列（SQ）或完成队列（CQ）损坏 | 删除并重建相关队列（删除 SQ 会中止所有未完成命令） |
| **控制器级严重错误** | 管理命令失败或队列删除无响应 | 执行控制器级重置（Controller Level Reset） |
| **致命错误** | `CSTS.CFS=1` 控制器无法报告错误 | 控制器重置；若失败且支持则使用 NVM 子系统重置（NSSR） |

**规范依据**: [PDF p. 690](../_source/pages/page-690.md)

### 关键不变式

1. **数据完整性**：对于介质错误（Media Error）、读取错误（Read Error）或内部错误（Internal Error），与失败命令相关的所有已传输数据都**不得**作为有效数据使用

2. **重置优先级**：
   - 优先使用控制器重置（Controller Reset）
   - 仅在控制器重置失败且系统支持时才考虑 NVM 子系统重置（NSSR）
   - ⚠️ **注意**：NSSR 可能导致 PCIe 链路断开（Link Down），对固件优先错误处理（Firmware First Error Handling）等环境不安全

**规范依据**: [PDF pp. 690-691](../_source/pages/page-690.md)

## 命令停止处理屏障

要安全地重试命令，主机必须首先确保控制器已经完全停止处理原命令。这个"停止屏障"由三个阶段构成：

### 停止屏障三阶段

| 阶段 | 主机操作 | 时间边界 | 说明 |
|-----|---------|---------|------|
| **1. 断开传输路径** | 对于基于消息的传输（Message-based Transport）：终止关联（Association）和所有传输连接 | 立即 | 基于内存的传输（Memory-based Transport）跳过此步骤 |
| **2. 等待控制器检测** | 等待控制器侧超时检测机制触发 | • **命令型检测**：从最后一次 Keep Alive 起等待 `2×KATT`<br>• **流量型检测**：从最后一条命令起等待 `3×KATT`<br>• **传输层通知**：接收传输特定的丢失通知 | 确保控制器已经感知到通信丢失 |
| **3. 等待命令停止** | 等待队列中的命令完全停止处理 | • 若 `CQT ≠ 0`：等待 CQT 时长<br>• 若 `CQT = 0`：等待可配置的实现特定时长 | 确保未完成命令不再被处理 |

**术语解释**：
- **KATT** (Keep Alive Timeout)：保活超时时间
- **CQT** (Command Queue Timeout)：命令队列超时时间
- **Association**：主机与控制器之间的逻辑关联

### 时间计算原理

`2×KATT` 和 `3×KATT` 的设计假设：
- 传输层最多延迟 **1 个 KATT** 周期
- 控制器检测超时需要 **1-2 个 KATT** 周期

### 特殊情况：后台操作

某些后台操作（如安全擦除 Sanitize）可以在通信丢失后继续运行。主机需要通过**其他控制器路径**跟踪这些操作的进度。

**规范依据**: [PDF p. 692](../_source/pages/page-692.md)

## 命令重试分类体系

NVMe 根据命令的幂等性和状态影响范围，将重试策略分为三类：

### 1. 无限制重试（Unrestricted Retry）

**适用条件**：
- ✅ 命令是**幂等的**（重复执行结果相同）
- ✅ 只改变**辅助状态**（Ancillary State），如统计计数器
- ✅ 命令效果标志位全为 0：`CCC=0, NIC=0, NCC=0, LBCC=0`

**恢复策略**：
在停止屏障建立后，可以**立即重试**，无需额外等待或检查。

**典型示例**：
- 读取不变数据的命令
- 查询状态的命令

---

### 2. 延迟重试（Delayed Retry）

**适用条件**：
- ✅ 命令是**幂等的**
- ⚠️ 会改变**用户数据**（User Data）或**子系统状态**（Subsystem State）
- ✅ **不会**影响其他主机

**恢复约束**：
在原命令及其之前的所有重试都**确认停止**之前，**不得**向上层软件报告最终结果。

**关键原则**：
即使命令本身幂等，也必须等待"原命令 + 所有重试"都不再可能执行后，才能暴露重试结果。这是为了防止破坏上层软件的命令顺序依赖。

**典型示例**：
- 单主机环境下对指定地址的写操作（Write）

---

### 3. 状态依赖重试（State-Dependent Retry）

**适用条件**（满足任一）：
- ❌ 命令是**非幂等的**（重复执行产生不同结果）
- ⚠️ 状态变化可能影响**其他主机**

**恢复策略**：
**不能**由 NVMe 驱动层自动重试，必须由**上层软件**（应用程序、文件系统、数据库）根据以下因素决策：
1. 原命令的实际处理程度
2. 当前子系统的状态
3. 正确的恢复动作

**典型示例**：
- 创建命名空间（Create Namespace）：可能已创建一个，重试会创建第二个
- 注销保留注册（Unregister Reservation）：原命令可能已成功，重试会失败
- 比较并写入（Compare and Write）：融合操作（Fused Operation）可能已修改数据
- 多主机环境下的写操作：可能覆盖其他主机的后续写入

**规范依据**: [PDF p. 693](../_source/pages/page-693.md)

---

### 幂等性判定标准

命令被视为**幂等**需满足：
- 在**无干预命令**的情况下重复提交
- 子系统的**终态**与**可观测结果**保持相同
- 辅助状态（如统计计数器）的变化**不破坏**幂等性

**幂等示例**：
- 对指定地址的读操作（Read）
- 对指定地址的写操作（Write）

**非幂等示例**：
- 创建命名空间（Create Namespace）
- 注销保留注册（Unregister Reservation）

---

### 保守策略

主机实现可以采用更保守的分类策略：
- 将 **无限制重试** 降级为 **延迟重试** 或 **状态依赖重试**
- 将 **延迟重试** 降级为 **状态依赖重试**

这样可以提高安全性，代价是降低恢复效率。

**规范依据**: [PDF pp. 694-695](../_source/pages/page-694.md)

## 常见误区与反例

### 误区1：停止屏障 = 可以盲目重试？

**错误理解**：只要建立了停止屏障，确认原命令不再处理，就可以安全重试所有命令。

**反例场景**（跨控制器路径）：

```text
时间线：
t1: 主机通过控制器1提交 Write A（写入值 A）
t2: 通信丢失，Write A 状态未知
t3: 主机在控制器2上建立停止屏障
t4: 主机在控制器2上重试 Write A → 成功 → 值变为 A
t5: 上层软件观察到 A，继续执行 Write B → 值变为 B
t6: ⚠️ 原 Write A 在控制器1上延迟完成 → 值退回到 A

最终结果：数据损坏，Write B 被覆盖
```

**正确理解**：
即使命令是幂等的，如果它会改变用户数据，就属于**延迟重试**，必须：
1. 确保原命令和**所有重试**都不再可能处理
2. 才能将重试结果暴露给上层软件

这是为了保护上层软件的命令顺序依赖（如 Write A → Write B）。

**规范依据**: [PDF pp. 702-703](../_source/pages/page-702.md)

---

### 误区2：重试成功 = 原命令失败？

**错误理解**：如果重试成功了，说明原命令没执行；如果重试失败了，说明原命令已执行。

**反例汇总**：

| 场景 | 原命令可能的结果 | 重试的观测结果 | 实际情况 |
|-----|----------------|---------------|---------|
| **创建命名空间** | 已成功创建 NS1 | 再次成功，创建 NS2 | 创建了两个命名空间 |
| **注销保留注册** | 已成功注销，不再是注册者 | 返回 `Reservation Conflict` 错误 | 原命令成功，重试失败 |
| **比较并写入** | 融合操作可能已修改数据 | 因比较条件改变而失败 | 数据已被修改 |

**正确理解**：
对于**状态依赖重试**，以下三个信息源**不能相互替代**：
1. 原命令的实际执行情况（未知）
2. 重试命令的返回状态（可观测）
3. 子系统的当前状态（需检查）

**规范依据**: [PDF p. 704](../_source/pages/page-704.md)

---

### 误区3：单主机环境就安全？

**错误理解**：只要系统中只有一个主机，就不会有状态冲突问题。

**反例场景**（多主机写入竞争）：

```text
主机1: 提交 Write A → 完成丢失 → 建立停止屏障 → 延迟重试 Write A
                 ↓
主机2:       观察到 A → 执行依赖操作 B → Write B
                                      ↓
                                  主机1 的重试 Write A 覆盖了 B

最终结果：主机2 的 Write B 被覆盖，数据不一致
```

**正确理解**：
即使主机1已建立停止屏障，其延迟重试仍可能覆盖其他主机在此期间的状态变更。因此：
- 会影响其他主机的命令属于**状态依赖重试**
- 普通的读写操作**不应**用作多主机协调机制
- 应使用专用机制：
  - **保留机制**（Reservations）
  - **比较并写入**（Compare and Write）

**适用范围**：
修改对其他主机可见的特性（Feature）或子系统状态也有同样的风险。

**规范依据**: [PDF p. 705](../_source/pages/page-705.md)

## 实践建议

### 驱动开发者

1. **优先建立停止屏障**：在任何重试之前，严格遵循三阶段停止屏障流程
2. **准确分类命令**：根据 CCC/NIC/NCC/LBCC 位和命令语义正确分类
3. **保守策略优先**：不确定时，采用更保守的重试分类
4. **日志记录**：记录通信丢失、停止屏障建立和重试决策的完整日志

### 系统架构师

1. **避免依赖重试**：将普通读写作为多主机协调机制是危险的
2. **使用专用机制**：多主机场景使用 Reservations 或 Compare and Write
3. **设计上层恢复**：对于非幂等操作，在应用层实现状态检查和恢复逻辑
4. **考虑后台操作**：设计系统时考虑后台操作可能穿越通信丢失继续执行

### 测试验证

1. **模拟通信丢失**：在测试环境中模拟各种时机的通信中断
2. **验证停止屏障**：确认停止屏障时间计算正确
3. **检查数据一致性**：在多路径和多主机场景下验证数据完整性
4. **压力测试**：在高负载下测试重试机制的正确性
