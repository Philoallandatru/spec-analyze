# 持久事件日志（Persistent Event Log）

## 概述

持久事件日志（Persistent Event Log）是 NVMe 子系统（NVM Subsystem）级别的重要事件历史记录机制，用于追踪系统运行过程中的关键事件。这些事件记录与具体命令的执行无关，而是反映子系统整体状态变化和重要操作的日志。

**核心特性：**
- **持久化存储**：事件记录在断电（power-off）和重置（reset）后仍然保留
- **全局可见**：整个 NVM 子系统共享同一份事件历史
- **快照读取**：通过冻结的报告上下文（reporting context）确保多次读取的一致性
- **时间有序**：通常按照最新事件优先的顺序返回

参考：[PDF pp. 252-255](../_source/pages/page-252.md)

## 事件记录的生命周期

### 从事件发生到主机读取

事件记录经历以下流程：

```text
事件发生 → 持久存储 → 报告上下文快照 → 主机读取
            ↓                  ↓
        后续事件保持在      代数检查
        快照之外          (generation check)
            ↓
        压力下由厂商删除/抑制
```

**流程说明：**

1. **事件发生**：控制器（Controller）检测到需要记录的事件
2. **持久存储**：事件被写入持久化存储介质
3. **快照创建**：主机建立报告上下文时，创建当前事件集合的快照
4. **主机读取**：主机通过多个命令分段读取快照内容
5. **一致性保障**：通过代数编号（generation number）检查确保读取完整性

### 事件记录策略

控制器对每个受支持的事件类型都会记录其发生，但存在以下限制机制：

**重复事件抑制：**
- 当某个事件短时间内重复发生，超过厂商定义的频率阈值（vendor frequency threshold）时，控制器可能不再记录后续重复事件
- 这样可以避免日志被重复事件淹没

**存储压力下的删除策略：**

当日志存储达到以下限制时，控制器会执行厂商特定的删除策略：
- **大小限制**：总字节数达到上限
- **数量限制**：事件条目数达到上限
- **类别限制**：特定类型事件达到分配限额

**多控制器事件处理：**
- 如果某个事件影响多个控制器，应由厂商选定的其中一个控制器记录该事件
- 避免同一事件被重复记录多次

**数据安全考虑：**
- 执行数据清除（Sanitize）操作时，可能会删除或修改某些事件记录
- 目的是防止从历史事件推断出用户数据的信息

参考：[PDF pp. 252-253](../_source/pages/page-252.md)

## 报告上下文（Reporting Context）管理

### 上下文生命周期

报告上下文是读取持久事件日志的核心机制，它提供了事件集合的一致性快照：

```text
无上下文 ──建立+读取──> 上下文存在 ──读取分片──> 释放上下文
   ↑                      ↓                       ↓
   └──重置/超时释放────────┘                  停止返回数据
```

**状态转换说明：**

| 状态 | 描述 | 允许的操作 |
|------|------|-----------|
| **无上下文** | 初始状态或上下文已释放 | 只能执行建立操作 |
| **上下文存在** | 已创建事件快照 | 可以读取、检查或释放 |
| **已释放** | 主动释放或超时释放 | 回到无上下文状态 |

### 上下文操作详解

| 操作类型 | 操作说明 | 行为约定 |
|---------|---------|---------|
| **Read（读取）** | 读取快照中的事件数据 | 要求上下文已经存在；否则返回命令序列错误（Command Sequence Error） |
| **Establish and Read<br>（建立并读取）** | 冻结事件长度和集合，然后返回请求偏移处的数据 | 如果上下文已存在则失败 |
| **Release（释放）** | 删除现有上下文 | 如果上下文存在则删除，不返回数据，幂等操作 |
| **Establish/Check Header<br>（建立/检查头部）** | 建立新上下文或检查现有上下文 | 始终返回 512 字节的头部信息，并报告上下文是否已经存在 |

**上下文自动释放条件：**

上下文会在以下任一事件发生时自动释放：
1. **显式释放**：主机主动执行释放操作
2. **NVM 子系统重置**：整个子系统执行重置
3. **控制器级重置**：单个控制器执行重置
4. **厂商超时**：超过厂商定义的超时时间（通常足够完成正常的检索操作）

**重要特性：**
- 上下文存在期间发生的新事件会被记录到持久存储，但**不会出现在当前快照中**
- 这确保了主机读取的一致性——同一个快照在整个读取过程中内容不变

参考：[PDF pp. 253-255](../_source/pages/page-253.md)

### 代数一致性协议（Generation Consistency Protocol）

对于需要分多次命令传输的大型日志，主机使用代数编号来验证数据完整性：

**一致性检查流程：**

1. **建立上下文**：主机发送建立命令，创建事件快照
2. **读取初始代数**：从头部获取代数编号（generation number）
3. **分段读取**：通过多个命令逐段读取事件数据
4. **验证代数**：读取完成后再次检查代数编号
5. **判断一致性**：
   - 代数编号一致 → 数据完整有效
   - 代数编号不一致 → 上下文可能已丢失，需要重新读取

**代数编号变化的含义：**
- 代数编号只在新建上下文且返回不同数据时才递增
- 如果两次读取同样的事件集合，代数编号不会改变
- 代数不一致意味着组装后的日志可能包含来自不同快照的数据片段，因此无效

参考：[PDF p. 254](../_source/pages/page-254.md)

## 日志结构

### 快照头部（Snapshot Header）

每个持久事件日志快照以 512 字节的头部开始，包含以下关键信息：

```text
┌─────────────────── 512 字节头部 ───────────────────┐
│ • 事件数量（event count）                          │
│ • 总字节长度                                       │
│ • 日志版本与头部版本（log/header revision）       │
│ • 上下文建立时间                                   │
│ • 电源历史（power history）                       │
│ • 控制器身份标识                                   │
│ • 子系统 NQN（NVM Qualified Name）                │
│ • 代数编号（generation number）                   │
│ • 报告上下文信息（Reporting Context Information） │
│ • 支持的事件位图（Supported Events Bitmap）       │
└───────────────────────────────────────────────────┘
│
├─ 事件 0
├─ 事件 1
├─ ...
└─ 事件 N
```

**报告上下文信息字段说明：**

| 信息项 | 内容 |
|--------|------|
| **上下文存在状态** | 指示上下文是否已经存在 |
| **建立端口类型** | 通过 NVM 子系统端口还是管理端点（Management Endpoint）建立 |
| **端口标识符** | 建立上下文的端口 ID |

**支持的事件位图：**

位图中的位编号直接映射到事件类型编号，涵盖以下事件族：

- **健康快照**：SMART/Health 状态定期记录
- **固件相关**：固件激活、更新事件
- **时间戳与重置**：时间变更、系统重置记录
- **硬件错误**：PCIe 错误、设备警告等
- **命名空间变更**：命名空间的创建、删除、修改
- **数据操作**：格式化（Format）、清除（Sanitize）操作
- **特性变更**：Feature 设置变更
- **遥测数据**：Telemetry 日志创建
- **温度事件**：过热、过冷、节流（throttling）
- **介质验证**：介质完整性检查结果
- **厂商特定事件**：厂商自定义事件
- **TCG 定义事件**：安全相关事件

参考：[PDF pp. 256-260](../_source/pages/page-256.md)

### 事件条目结构（Event Entry Framing）

每个事件条目采用自描述格式，可以独立解析：

```text
┌─────────── 事件头部 ───────────┐
│ • 事件类型（event type）       │
│ • 事件版本（revision）         │
│ • 头部长度（EHL）              │
│ • 控制器 ID                    │
│ • 时间戳                       │
│ • 端口关联                     │
│ • 厂商信息长度（vendor-info）  │
│ • 事件数据长度（EL）           │
└───────────────────────────────┘
│
├─ 可选厂商信息（vendor information）
│
└─ 事件类型特定数据（type-specific event data）
```

**解析机制：**

- **头部长度**：`EHL + 3` 给出事件头部的总字节数
- **事件长度**：`EL` 涵盖可选厂商信息和类型特定数据的总长度
- **独立版本控制**：事件类型版本（Event Type Revision）独立于整体日志版本，允许特定事件类型单独演进

**端口关联类型：**
- 子系统端口（subsystem port）
- 管理端点（Management Endpoint）
- 无端口关联（no port）

**版本支持说明：**

支持持久事件日志的 NVMe 2.1 版控制器会公布 `ECRH`（Establish Context and Read Header）能力位，表明支持 512 字节的建立/检查头部操作以及代数编号一致性协议。

参考：[PDF pp. 258-260](../_source/pages/page-258.md)

## 标准事件类型

### 基础事件记录

这些事件是控制器必须或在特定条件下必须记录的：

| 事件类型 | 记录内容 | 记录频率/条件 |
|---------|---------|--------------|
| **SMART/Health 快照** | 完整的 512 字节 SMART/Health 信息镜像 | • 每个控制器至少每 24 个通电小时记录一次<br>• 在虚拟化管理下，每个主控制器（primary controller）记录一次 |
| **固件提交<br>（Firmware Commit）** | • 旧/新固件版本<br>• 操作动作<br>• 固件槽位<br>• 完成状态码（SCT/SC）<br>• 可选的厂商结果 | 固件提交操作时记录 |
| **时间戳变更** | • 先前的时间戳<br>• 自控制器级重置以来的毫秒数<br>*当前时间戳在通用事件头部中* | 通过 Set Features 命令修改时间戳时记录 |
| **上电或重置** | • 生效的固件版本<br>• 每个控制器的重置描述符 | • 每次上电或重置<br>• 虚拟化环境下每个主控制器记录 |

**重置描述符包含的信息：**
- 控制器 ID
- 固件激活成功或失败状态
- 是否正在执行 Format NVM 操作
- 控制器上电循环次数
- 上电毫秒数
- 控制器时间戳

这些强制和条件记录保留了足够的跨控制器数据，使得主机可以重建完整的固件与重置历史。

参考：[PDF pp. 260-262](../_source/pages/page-260.md)

### 硬件错误事件

硬件错误事件（NVM Subsystem Hardware Error）使用两字节的错误代码，后跟代码特定的附加信息。

**事件标识：**
- 事件类型：`05h`
- 事件类型版本：`02h`

**错误分类与附加信息：**

```text
硬件错误代码
  ├─ PCIe 可纠正错误 ──────> 设备状态、AER 状态、掩码、头部日志与前缀日志
  ├─ PCIe 不可纠正非致命错误 ──> 同上
  ├─ PCIe 不可纠正致命错误 ──> 同上
  ├─ PCIe 链路状态变更 ───> PCIe 链路状态寄存器
  ├─ PCIe 链路未激活 ─────> 无附加信息
  ├─ 警告事件 ──────────> 警告位图 + 可选的耐久组 ID
  ├─ 意外断电 ──────────> 生命周期断电次数 + 带外原因标志
  ├─ 致命控制器状态 ─────> 状态信息
  ├─ 介质/数据完整性错误 ──> 捕获的完成队列条目（CQE）
  └─ 控制器就绪超时 ─────> CNR 状态 | Admin 介质未就绪 | 命名空间未就绪 | CRIME
```

| 错误类别 | 附加信息内容 |
|---------|-------------|
| **PCIe 错误**<br>（可纠正/非致命/致命） | • 设备状态寄存器<br>• AER（Advanced Error Reporting）状态<br>• 错误掩码<br>• TLP（Transaction Layer Packet）头部日志<br>• TLP 前缀日志<br>*注：仅在支持 PCIe AER 时包含* |
| **PCIe 链路状态变更** | 捕获事件发生时的 PCI Express 链路状态寄存器 |
| **PCIe 链路未激活** | 无附加硬件错误信息 |
| **临界警告** | • 警告位图（warning bitmap）<br>• 耐久组标识符（Endurance Group ID，如适用） |
| **意外断电** | • 生命周期断电次数计数器<br>• 带外断电原因标志（out-of-band cause flag） |
| **致命控制器状态** | 控制器状态信息 |
| **介质或数据完整性错误** | 捕获的完成队列条目（CQE），包含错误状态码 |
| **控制器就绪超时** | • 控制器未就绪（CNR）状态<br>• Admin 命令介质未就绪标志<br>• 命名空间未就绪标志<br>• 所选的介质独立就绪模式（CRIME） |

**重要规则：**
- 除非错误代码特别说明，不可用的可选信息字段应清零
- 子系统会记录其厂商支持集合中检测到的每个错误
- 遵循通用的重复事件抑制规则

附加结构保留了事件发生时的状态。PCIe AER 字段仅在支持 AER 时存在；ready-timeout 字节区分命令处理就绪与附加命名空间、Admin 命令的介质就绪，并记录所选的介质独立就绪模式。

参考：[PDF pp. 263-266](../_source/pages/page-263.md)

### 命名空间与破坏性操作日志

#### 命名空间变更事件

Change Namespace 事件记录成功的命名空间管理操作的输入参数和结果：

**记录的信息：**
- 大小和容量（size/capacity）
- 格式与保护属性（format/protection）
- 共享属性（sharing）
- ANA 组（Asymmetric Namespace Access Group）
- NVM 集合（NVM Set）
- 耐久组（Endurance Group）
- 创建或删除的命名空间 ID（NSID）

**特殊情况：**
- 删除所有命名空间的操作使用 `FFFFFFFFh` 作为 NSID
- 每个命名空间的属性字段保持保留状态

参考：[PDF pp. 266-268](../_source/pages/page-266.md)

#### 破坏性操作的配对记录

破坏性操作使用"开始"和"完成"配对事件，记录操作的意图和最终结果：

| 操作类型 | 开始事件记录 | 完成事件记录 |
|---------|-------------|-------------|
| **Format NVM<br>（格式化）** | • 已验证的命名空间 ID<br>• 控制器格式属性<br>• 命令 Dword 10 参数<br>*记录时机：介质修改前* | • 观察到的最小剩余进度<br>• 未完成/错误标志<br>• 完成队列条目状态<br>• 厂商完成信息 |
| **Sanitize<br>（清除）** | • 能力信息<br>• 命令 Dword 10-11 参数<br>*记录时机：目标进入处理状态时* | • 清除进度/状态<br>• 厂商完成信息<br>*记录时机：目标达到空闲或失败状态时* |

**配对记录的价值：**
- 区分"已接受的操作意图"与"最终执行结果"
- 记录部分完成的格式化操作
- 记录清除操作的失败情况
- 便于诊断操作失败的原因

参考：[PDF pp. 268-269](../_source/pages/page-268.md)

### Feature 变更日志

Set Feature 事件在受支持的、可记录的 Feature 通过 Set Features 命令成功变化时是**必须记录**的。

**记录规则：**
- 成功的配置变更必须记录
- 成功的无操作（no-op）设置也可以记录
- 记录资格因 Feature 类型和控制器类型而异
- 某些 Feature 与控制器类型的组合被禁止或不推荐记录

**事件载荷内容：**

可变长度的事件载荷标识以下信息：
- 从 Dword 10 开始保留了多少个连续的命令 dword
- 可选的完成 Dword 0（completion Dword 0）
- 内存缓冲区字节数（memory-buffer bytes）

这使得不同 Feature 的变更事件可以根据需要携带不同长度的参数数据。

参考：[PDF pp. 270-271](../_source/pages/page-270.md)

### 诊断与扩展事件

| 事件类型 | 持久化证据内容 |
|---------|--------------|
| **Telemetry Created<br>（遥测数据创建）** | 主机发起（Host-Initiated）或控制器发起（Controller-Initiated）遥测日志的第一份 512 字节头部副本 |
| **Thermal Excursion<br>（温度异常）** | 阈值转换类别，加上与跨越阈值的绝对开尔文温度差 |
| **Sanitize Media Verification<br>（清除介质验证）** | 当清除操作进入介质验证阶段时的仅头部标记 |
| **Vendor-specific<br>（厂商特定）** | • 一份或多份类型化描述符<br>• 稳定的事件代码<br>• UUID 索引<br>• 名称/字符串/二进制/有符号整数数据 |
| **TCG-defined<br>（TCG 定义）** | 事件版本和保留给 TCG 的载荷 |

#### 温度事件详细分类

温度相关事件区分以下具体情况：

**过热事件：**
- 高温警告阈值跨越（high warning threshold）
- 临界温度阈值跨越（critical threshold）
- 节流开始（throttling start）
- 节流停止（throttling stop）
- 返回正常温度（return to normal）

**低温事件：**
- 低于低温阈值

**注意事项：**
- 即使是重复的阈值跨越报告，仍然受到厂商频率抑制规则约束
- 避免温度频繁波动时产生过多日志条目

参考：[PDF pp. 272-275](../_source/pages/page-272.md)

#### 厂商特定事件结构

厂商特定事件使用描述符机制，支持灵活的数据类型：

```text
厂商特定事件
  └─ 描述符列表
      ├─ 事件代码 + UUID 索引
      └─ 类型化数据
          ├─ 名称（name）
          ├─ ASCII 字符串
          ├─ 二进制数据（binary）
          └─ 有符号整数（signed integer）
```

**UUID 索引过滤：**
- 当 Get Log Page 命令指定 UUID 索引时
- 返回的厂商事件包含该 UUID 的事件
- 同时包含由子系统制造商定义的通用事件

**命名稳定性：**
- 如果事件名称描述符存在，它排在描述符列表首位
- 同一厂商事件代码下的名称保持稳定，便于解析

参考：[PDF pp. 273-275](../_source/pages/page-273.md)

### 控制器重置

[Controller Enable, Shutdown, and Reset](controller-enable-shutdown-reset.md) 定义了可以释放报告上下文的重置边界：
- NVM 子系统重置会释放上下文
- 控制器级重置也会释放上下文
- 但存储的事件本身会保留，不受重置影响

参考：[PDF pp. 135-141, 252-253](../_source/pages/page-135.md)
