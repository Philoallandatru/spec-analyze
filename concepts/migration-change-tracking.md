# Migration Change Tracking（迁移变更追踪）

## 概述

迁移变更追踪（Migration Change Tracking）是 NVMe 规范中的一项功能，用于在控制器迁移过程中追踪数据变更。它通过一对配合使用的命令——Track Send（发送追踪）和 Track Receive（接收追踪）来实现控制平面的变更记录。

该功能可以追踪两类变更：
- **命名空间用户数据的变更**：记录写入操作到用户数据迁移队列（User Data Migration Queue）
- **主机内存的写入**：记录由指定控制器引起的主机内存写入操作

> **规范参考**：[PDF pp. 433-438](../_source/pages/page-433.md)

## 工作原理

### 交互模型

下图展示了管理控制器和目标控制器之间的交互流程：

```text
管理控制器 (Management Controller)          目标控制器 (Target Controller)
       |                                              |
       |  Track Send: 指定追踪范围                     |
       |--------------------------------------------->|
       |                                              | 命令处理并写入主机内存
       |                                              |
       |  追踪状态：记录已变更的范围                   |
       |<---------------------------------------------|
       |                                              |
       |  Track Receive: 请求获取变更描述符             |
       |--------------------------------------------->|
       |                                              |
       |  返回描述符（成功读取后即被消费）              |
       |<---------------------------------------------|
       |                                              |
       |- MTR=1: 需要重复执行 Receive 获取更多变更     |
       |- MTR=0, SUSP=1: 目标控制器挂起时完成          |
```

### 关键概念说明

- **处理追踪的控制器**：负责执行 Track Send/Receive 命令的控制器
- **被观测的控制器**：其写入操作被追踪记录的目标控制器
- **破坏性读取**：Track Receive 成功返回后，已报告的变更记录会被清除
- **挂起终止规则**：当目标控制器挂起时，可以确保不会再有新的变更产生

> **规范参考**：[PDF pp. 434-438](../_source/pages/page-434.md)

## 追踪模式

NVMe 支持两种追踪模式，通过 Track Send 命令的 `SEL` 字段选择：

### 模式对比表

| 追踪模式 | SEL 值 | 追踪的变更类型 | 变更记录方式 | 适用场景 |
|---------|--------|---------------|-------------|---------|
| **用户数据变更日志** | `SEL=0h` | 对命名空间（Namespace）的用户数据修改操作 | 控制器向控制器数据队列（CDQ）发布条目，条目格式由 I/O 命令集定义 | 追踪命名空间级别的数据变更 |
| **主机内存变更追踪** | `SEL=1h` | 由指定目标控制器引起的主机内存写入 | 控制器记录主机内存地址范围的变更 | 追踪控制器对主机内存的直接写入 |

### 模式启动与停止

- **用户数据日志模式**：通过 `LACT` 字段启动或停止，使用控制器本地的 `CDQID`（控制器数据队列 ID，Controller Data Queue ID）标识队列
- **主机内存追踪模式**：通过 `TACT` 字段启动或停止，使用目标控制器的 `CNTLID`（控制器 ID，Controller ID）标识目标

> **规范参考**：[PDF pp. 436-438](../_source/pages/page-436.md)

## 主机内存追踪详解

### 追踪参数配置

启动主机内存追踪时，需要在 Track Send 命令中提供以下参数：

| 参数 | 说明 | 约束条件 |
|-----|------|---------|
| **内存范围** | 一个或多个需要追踪的对齐内存范围 | 地址必须按粒度对齐 |
| **粒度** | 追踪的最小单位，编码为 `2^n × 4 KiB` | 必须在 Identify Controller 返回的最小/最大粒度范围内 |
| **描述符数量** | 能够记录的变更描述符数量 | 控制器本地和子系统级别都有数量限制 |
| **长度限制** | 可选：要求每个范围的长度必须是 2 的整数次幂 | 如果启用此规则，所有范围长度必须符合要求 |

> **规范参考**：[PDF pp. 437-438](../_source/pages/page-437.md)

### 追踪启动的限制条件

为了保证追踪的正确性和避免冲突，系统对追踪启动设置了以下限制：

1. **不能自我追踪**：处理追踪命令的控制器不能追踪自己
2. **不能追踪挂起的目标**：不能对已经处于挂起状态的目标控制器启动追踪
3. **避免重复追踪**：如果子系统中已有其他控制器正在追踪某个目标，则会拒绝新的追踪请求
4. **停止追踪**：将 `TACT` 清零可以停止追踪，此时数据缓冲区会被忽略

> **规范参考**：[PDF p. 438](../_source/pages/page-438.md)

### 请求描述符格式

追踪请求使用以下结构：

- **版本**：版本 0
- **描述符计数**：基于 1 的计数方式（1 表示第一个描述符）
- **描述符大小**：每个描述符 12 字节
- **描述符内容**：
  - 起始地址：按页粒度对齐的主机内存地址
  - 长度：以页为单位的长度
  - 用途：标识目标控制器通过 PRP（物理区域页，Physical Region Page）或 SGL（分散/聚集列表，Scatter/Gather List）访问的主机内存地址

**无效情况**：
- 地址或长度未按要求对齐
- 地址不在目标控制器关联的主机内存范围内

> **规范参考**：[PDF p. 439](../_source/pages/page-439.md)

### 资源协商机制

如果控制器无法满足请求的粒度或描述符数量：

1. **返回错误**：返回 "Not Enough Resources"（资源不足）状态
2. **提供实际能力**：在响应中附带控制器实际支持的粒度和描述符数量
3. **主机重新请求**：主机可以根据实际能力调整参数后重新发起请求

### 特殊情况和注意事项

- **内存重新配置**：如果后续的系统配置将已接受的追踪范围从主机内存变更为非主机内存，则后续追踪行为未定义（应避免这种情况）
- **MSI/MSI-X 寄存器**：不应请求追踪 MSI/MSI-X 中断寄存器范围，因为控制器报告这些地址的写入是可选的，可能不会被追踪

> **规范参考**：[PDF pp. 439-440](../_source/pages/page-439.md)

## 变更接收与消费

### Track Receive 命令参数

- **选择类型**：固定选择 "Tracked Memory Changes"（已追踪的内存变更）
- **请求大小**：使用基于 0 的 dword 计数（0 表示 1 个 dword）

### 消费语义

Track Receive 命令遵循"破坏性读取"语义：

- **成功读取后清除**：成功的 Receive 操作会移除（消费）已报告的变更记录
- **累积报告**：每次报告覆盖自追踪开始或自上次成功报告以来发生的所有变更
- **公平调度**：对同一地址的高频变更可能会被延迟报告，以确保其他地址的变更也能被公平地报告

> **规范参考**：[PDF pp. 433-434](../_source/pages/page-433.md)

### 响应数据格式

#### Header（响应头）

| 字段 | 说明 |
|-----|------|
| **Target Controller ID** | 目标控制器的标识符 |
| **粒度** | 变更报告使用的粒度 |
| **描述符数量** | 本次返回的变更描述符数量 |
| **MTR** | More To Receive 标志 |
| **SUSP** | 目标控制器是否在命令处理期间被挂起 |

#### MTR 标志含义

- **MTR = 1**：还有更多变更等待接收，需要再次执行 Track Receive 命令
- **MTR = 0**：处理时刻没有更多可用的变更记录

#### MTR 和 SUSP 组合的特殊意义

当 `MTR=0` 且 `SUSP=1` 时，建立了一个重要的保证：
- 目标控制器已被挂起
- 在目标控制器恢复之前，不会再产生新的已追踪变更
- 这个状态为迁移操作提供了一个确定的截止点

> **规范参考**：[PDF pp. 434-435](../_source/pages/page-434.md)

#### 变更描述符

每个描述符占用 16 字节，包含：

| 字段 | 说明 |
|-----|------|
| **起始地址** | 按报告粒度对齐的变更起始地址 |
| **长度** | 以报告粒度为单位的变更长度 |

**特殊情况**：零个描述符必须对应 `MTR=0`（表示没有变更可报告）

> **规范参考**：[PDF p. 435](../_source/pages/page-435.md)

## 用户数据日志模式

### 启动条件检查

启动用户数据迁移队列日志记录时，系统会进行以下检查：

| 检查项 | 失败条件 |
|-------|---------|
| **队列存在性** | 所选的控制器数据队列在处理命令的控制器上不存在 |
| **重复记录** | 已有其他控制器正在为该队列关联的目标控制器记录日志 |
| **目标状态** | 目标控制器处于挂起（Suspended）状态 |

如果任何一项检查失败，Track Send 命令将返回错误。

### 日志条目格式

- 日志条目的格式由所选的 **I/O 命令集**（I/O Command Set）定义
- 不同的 I/O 命令集可能有不同的条目格式和发布要求
- 控制器将变更信息发布到指定的控制器数据队列（CDQ）

> **规范参考**：[PDF pp. 436-437](../_source/pages/page-436.md)

## 相关概念关联

理解迁移变更追踪需要了解以下相关概念：

### Controller Migration（控制器迁移）

- 提供了挂起边界，使得最终的 `MTR=0` 报告具有确定性
- 挂起机制确保在迁移的最后阶段不会有新的变更产生
- **参考文档**：[Controller Migration](controller-migration.md)
- **规范参考**：[PDF pp. 434-435](../_source/pages/page-434.md)

### Controller Data Queues（控制器数据队列）

- 定义了用户数据迁移队列的创建方式
- 管理队列的内存生命周期
- 提供控制器本地标识符（CDQID）
- **参考文档**：[Controller Data Queues](controller-data-queues.md)
- **规范参考**：[PDF pp. 436-437](../_source/pages/page-436.md)

### Identify Command Model（识别命令模型）

- 披露控制器支持的内存范围粒度（最小值和最大值）
- 披露描述符容量限制（控制器本地和子系统级别）
- 这些信息用于追踪启动时的参数配置和验证
- **参考文档**：[Identify Command Model](identify-command-model.md)
- **规范参考**：[PDF pp. 363-364, 437-438](../_source/pages/page-363.md)

## 应用场景

迁移变更追踪在以下场景中发挥重要作用：

1. **虚拟机热迁移**：追踪虚拟机使用的存储变更，支持在迁移过程中进行增量数据同步
2. **存储控制器故障切换**：在控制器切换过程中追踪数据变更，确保数据一致性
3. **存储迁移和负载均衡**：追踪活动数据变更，支持在线的存储迁移操作
4. **一致性检查点**：通过挂起机制建立明确的一致性点，便于创建快照或备份
